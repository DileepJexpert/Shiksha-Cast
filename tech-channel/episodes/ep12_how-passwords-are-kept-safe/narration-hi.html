EP12 — How are passwords kept safe?

01Title

02The Hook

Ek sach mein wild baat. Jab website theek se bani ho, to wo aapka actual password kabhi nahi jaanti. Na staff, na database, na wo log jinhone use banaya — koi bhi use dekh nahi sakta. To agar wo password dekh hi nahi sakte… to ye check kaise karte hain ki aapne sahi type kiya?

[CUE] Yahan energy high — ye hook hai. "koi bhi" pe zor.
03Aaj ka sawaal

Toh aaj ka ek sawaal: agar website aapka password store hi nahi karti, to ye check kaise karti hai ki aapne sahi daala? Chalo pata karte hain.

[CUE] Thoda slow. Sawaal ko land hone do, phir aage.
04Idea 1 — Store nahi karte

Idea number one: wo aapka password store karte hi nahi. Jab aap sign up karte ho, site aapka chuna hua password save nahi karti. Iske badle wo ek hash save karti hai — aapke password ka ek scrambled "fingerprint". Soch lo jaise password se bana ek taala: taala aap password se bana sakte ho, par taale se wapas password nahi nikaal sakte.

[CUE] "fingerprint" pe zor — taale wali analogy pe card-note reveal karo.
05Idea 2 — Hashing one-way hai

Idea do: hash function one-way hota hai. Koi bhi input do — jaise "password123" — aur ye use ek fixed-length jumble bana deta hai, jaise "ef92a7c4d1…". Aage jaana easy hai. Par peeche jaana — jumble ko wapas password banana — practically impossible hai. Aur ye important baat: ye encryption nahi hai. Encryption reversible hai agar key ho. Hash jaan-boojhke one-way hai — yahi to pura point hai.

[CUE] Code ka arrow aage padho, phir "wapas nahi" wali line pe ruko.
06Idea 3 — Login = hashes compare

Toh agar password store hi nahi hota, to login kaam kaise karta hai? Simple — hashes compare karke. Jab aap login karte ho, site aapke abhi type kiye password ko leti hai, uska hash banati hai, aur use pehle se saved hash se compare karti hai. Agar dono hashes match ho gaye, to password sahi tha. Verify ho gaya ki ye sach mein aap ho — bina kabhi aapka real password store ya dekhe.

[CUE] queue ko left-to-right trace karo; "match" pe compare dikhao.
07Idea 4 — Salting

Ek aur clever trick: salting. Hash karne se pehle, site har password mein ek unique random value — ek "salt" — add kar deti hai. Toh agar do log bilkul same password chunte hain, phir bhi unke hashes alag aate hain. Isse pre-computed attack tables, jinhe rainbow tables kehte hain, bhi toot jaate hain — kyunki attacker us salt ke liye tayyari kar hi nahi sakta jo usne dekha hi nahi. Har ek password ko apna alag salt milta hai.

[CUE] Dono "hello" lines alag hash dete dikhao; "alag" pe zor.
08Why it matters

Aap ko isse farak kyun padta hai? Teen real reasons. Ek: jab koi achhi company breach hoti hai, attackers ko sirf hashes milte hain — aapka actual password nahi. Do: agar koi site aapko aapka apna password email kare, to ye ek bada red flag hai — matlab unhone use plain text mein store kiya hai. Aur teen: kabhi same password alag-alag sites pe reuse mat karo, kyunki ek leak se baaki sab kuch unlock nahi hona chahiye.

[CUE] Teen points ungliyon pe gino; "red flag" pe punch.
09Common myth

Aur isse aata hai sabse bada myth. Log sochte hain website unka password store karti hai — shayad encrypted — aur chahe to use wapas padh sakti hai. Par theek se bani site ek one-way salted hash store karti hai. Wo aapko verify kar sakti hai, par aapka actual password kabhi recover nahi kar sakti. Difference yaad rakho: encryption key se undo ho sakti hai — hash nahi. Yahi pura magic hai.

[CUE] Myth pe ek beat ke baad green "Reality" card reveal karo.
10Recap

Quick recap. Ek: passwords one-way, salted hashes ki tarah store hote hain — kabhi real password nahi. Do: login hashes compare karke kaam karta hai, isliye aapka real password kahin store hi nahi hota. Verify, par store kabhi nahi.

[CUE] Crisp aur clear — ye do lines log yaad rakhein.
